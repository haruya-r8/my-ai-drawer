## 1. 要件定義・スコープ管理

### 小さく産んで大きく育てる（ミニマムスタートの原則）
* **運用の代替案も視野に入れる**: 1つの目的を達成するために複数システムの操作を強いる設計は避けるべきだが、すべてをシステム化するとコストが膨らむ。何を機能（システム）化し、どこを運用でカバーするかの境界線を厳密に見極める。
* **実装を見据えた要件定義**: 要件定義の段階から、最終的な実装（実現可能性やコード構造）を想定してスコープを決定する。

### ユーザーは「要望」は出せるが「要件」は定義できない
* **提示と合意形成のプロトコル**: ユーザーに「システム要件」をゼロから求めても出てこない（例：一般ユーザーに次世代OSの要件を求められないのと同様）。開発側がシステム要件定義としてドキュメント化し、「これで要件を満たせるか」を確認してもらうアプローチを徹底する。
* **システム要件定義の流動性**: 業務要件（ユーザーの使い勝手や業務フロー）は後から変更すべきではないが、システム要件（実現手段）は設計フェーズで不都合が生じた場合、こだわりすぎずに柔軟に修正・最適化してよい。

## 2. 設計・リファクタリング（コードとデータの設計）

### 抽象化と共通化のトリガー
* **コピー＆ペーストの禁止**: コードをコピーして流用したくなった時点が、共通化を検討すべきシグナルである。共通化を怠ると、将来の修正時に二重改修が発生し、高確率で修正漏れの原因となる。
* **コードレビュー時の広域視点**: 目前の修正による既存への影響（デグレ）を過度に恐れず、システム全体のコード共通化・クリーン化を常に模索する。
* **関数命名は「機能」ベースで行う**: 特定の画面やユースケース専用の命名（例: `submitUserForm`）にすると再利用性が下がる。純粋な機能名（例: `validateInputData`）にすることで、他コンポーネントからの共通利用を促す。

### 可読性とセキュリティのバランス
* **過剰なコメントの排除**: コメントが多すぎるコードは逆に可読性を損なう。コードそのものの自己記述性を高め、コメントの残し方はプロジェクトの標準ルールに準拠させる。
* **SQLリテラルの許容**: すべてを定数化すると、かえてコードの可読性が下がるケースがある。SQLのリテラル表記など、定数化の費用対効果が低いものはそのまま記述してよい。
* **安易なDB外だし化への警戒**: 「簡単に変更できるように」と何でもDBのマスターデータ管理に逃げると、データ改ざんや不正アクセスのリスクなど、セキュリティ面の脆弱性を生む原因になる。方針検討時は慎重に見極める。

## 3. UI/UX・ユーザー視点の設計

### 「正確さ」より「伝わりやすさ」を優先する
* **システム視点からの脱却**: 開発者にとってロジカルに正しい（例外のない）表現が、ユーザーにとっても分かりやすいとは限らない。
* **ステータス表現の最適化**: 例えば「予約済み」「未予約あり」といった表記は、システム視点では正確でも、ユーザーには直感的に伝わりにくい。「済」か「未」かが一目で判別できるシンプルさが重要。また、「抽出」や「除外」といったデータの処理方向も、ユーザーの認知に合わせた言葉選びを徹底する。

## 4. テスト設計・品質保証

### テストフェーズの定義と環境整備
* **IT（結合テスト）とST（システムテスト）の境界**: 
  * **IT（結合テスト）**: 自システム内のモジュール・機能間に閉じたテスト。この段階では連携先システムの結果まで追わない。
  * **ST（システムテスト）**: 連携先システムまで含めた、エンドツーエンド（E2E）の結果整合性を確認する。
* **テスト環境の妥協禁止**: 開発・テスト環境の制約により確認できない項目がある場合、「仕様がない」と諦めるのではなく、テストが実行できるよう環境を整備・ハックするアプローチを取る。

### テスト観点と評価基準
* **設計段階での実装・テスト想定**: 製造（コーディング）からの手戻りを最小化するため、基本・詳細設計の段階で実装およびテストのしやすさを徹底的にシミュレートする。要件定義の漏れは必ず発生するという前提に立つ。
* **データコンペア（比較）による保証**: 差分連携のシステムテストなどでは、最終的に全件データが一致するかを突合（コンペア）する観点を取り入れ、システムの信頼性を担保する。
* **単体テスト（UT）と結合テスト（IT）の役割分担**: 
  * 条件分岐網羅（ロジック）は単体テストで担保する。
  * 単体テストをパートナーや別担当に一任している場合は、レビュー権限のある結合テストのシナリオにあえて網羅項目を記述しておき、確認漏れを防ぐ防衛策をとる。
* **UT（単体テスト）の合格基準**: 単体テストの本質は「要件（作りたいもの）が正しく実装されているか」の検証である。実装者の「やりたいこと（意図）」がレビュー側に伝わらない設計・コードは、それだけでNG（不合格）と判定する。
* **品質報告の罠（数値の盲信禁止）**: メトリクス（バグ密度・テスト密度など）が管理値の範囲内に収まっているという「結果」だけで品質OKと判断してはならない。適切なテストケースが選択され、妥当なバグ収束曲線を描いているかという「プロセス（プロセス品質）」を踏まえた上で判断する。

## 5. PM・プロジェクトマネジメント（計画と管理）

### WBSと見積りのコントロール
* **1タスクは5日（1週間）以内に分解する**: 週次で進捗管理を行うため、タスクの粒度が5日を超えると進捗の遅れ（スタック）を検知できない。見積り時は、過去の類似実績をベースに、5日以内に収まる粒度までブレイクダウンする。
* **WBS作成後のリソース検証**: WBSを引いた後、必ず週間工数（人日/人月）に逆算して引き直し、メンバーの稼働負荷が現実的（妥当）であるかを確認する。
* **進捗評価の単純化（傾斜なし管理）**: 進捗率を「20%」「50%」と細かく見積もるのではなく、「未着手（0）か完了（100）か」の2択（または0/50/100）で、実施したか否かだけを定量的に測る手法も有効。

### 方針変更へのマインドセット
* **プロトタイプ（デモ）要望への先回り**: 「とりあえずデータが送られて見られれば良い」というスモールスタートで始まった案件が、途中で「デモを見せたい（リッチにしてほしい）」と方針変更されるのは日常茶飯事である。
* **教訓**: 最初のヒアリング時点で「後々、分析やデモができる準備（拡張性）を考慮しておかなくてよいか」を開発側から先回りして提案（逆提案）すべきであった。次回以降の要件定義にこの観点を活かす。

### テクニカルデット（技術負債）の解消
* **設計書の計画的アップデート**: 既存システムの設計書が形骸化している、あるいは存在しないケースは多い。そのまま放置すると次回EOL（製品サポート終了）時に全く同じ苦労を再現することになる。プロジェクトのライフサイクル内で、計画的に設計書をリバイズ（再整備）していくマインドを持つ。

## 6. 原文（インプットデータ）

> ### システム開発の学び2023-2024
> 
> ◼️2023年8月
> * コメントを多くしすぎない：可読性が悪くなる。残し方はプロジェクトに応じて検討する。
> * コードレビュー指摘：既存への影響もあるが共通化を考える。
> * 方針検討：DBに持って簡単に変更できるとセキュリティがあやしくなる。
> 
> ◼️2023年9月
> * １つのことをするのに複数のシステムの操作をさせない。必要かもしれないがコストもかかるので、何を機能としてどこを運用で対応するか考える。小さく産んで大きく育てる。実装まで考えて要件定義する。
> * 方針変更を想定する：BDの件、とりあえず送って見れるようにしておいてで始まったのに、デモできるようにって。まぁでも前向きに考えたらそうだよなと思う。最初に聞いたときそれでいいのかと思ったし。分析できる準備はしておかなくてよいでしょうかと聞いた方が良かったよな。次からできたらいいな。
> * ITとST：IT（結合テスト）は自システムに閉じたテスト。連携先の結果を見ない。ST（連携テスト）は連携先のシステムの結果まで確認する。
> * 設計で実装を想定する：製造の戻りを少なくするため、設計でしっかり実装を想定する。要件定義に不足不備は少しはあるもの。
> 
> ◼️2023年10月
> * システム要件定義には拘らない：業務要件定義は使い勝手に関わるため、直さない方がよいが、設計で不都合が出てきたらシステム要件定義はあまり拘らずに修正する。
> * 見積り、計画の粒度は1タスク5日以内：週次で進捗管理するので、5日以内にしておかないと管理できない。見積りは大体これがあるからこのぐらいという感じにしていく。傾斜を0にしてしたかどうかだけ測るのもあり。
> * 設計書を計画的に整備する：設計書が整備されてないのはよくあること。そのままだと次回EOLで同じように困るので計画的に整備していく考え方もある。
> * ユーザー視点で画面設計：詳しく正確に表現することが正しいわけではない。伝わることが大事。「予約済み」「未予約あり」がわかり辛かった。システム視点で見る人は何も無かったのに。言われてみるとわかり辛いし、自分でもなんか違和感あった部分。済か未がわかったらよい。そうだよなーと衝撃を受けた。抽出なのか除外なのかも伝わって無かった。
> * 要件はユーザーに確認してもらう：要望は出てくるが要件は出てこない。ドキュメントを作る必要があるから。例えばWindows12の要件を出せるかといわれると出せない。それと同じ。システム要件定義を作り、これでよいかと確認する。
> * 開発環境のテスト：開発環境のテストで出来ないことがある時は、できないから仕方ないではなくてテストできるように環境を整えることを考える。
> * 関数は機能名にする：何かの専用の関数名にしてしまうと他で使いにくい。機能名にしておけば、他からも使える。
> * SQLのリテラルはOK：定数にすると可読性が下がる。
> * コピー作成時は共通化を検討する：そうしないと、修正時に両方修正になり漏れる。
> 
> ◼️2023年11月
> * システムテストにコンペアの観点：どうやったら保証できるかを考えないといけない。差分連携するので、最後に全件合うか確認は良いと思った。
> * 単体テストと結合テスト：条件分岐のテストを単体でやっているなら、結合でやらなくてもいい。ただ今は単体は任せているので、レビューする結合テストに書いておいて確認を漏れないようにする。
> * WBS作成後の確認：週次の工数に直して、妥当か確認する。
> 
> ◼️2023年12月
> * UTの基準：要件の確認なので、やりたい事が伝わっていなかった以外はNG
> 
> ◼️2024年8月
> * 品質報告の基準値注意事項：範囲内というだけでOKではない。プロセスを踏んだ上で範囲内だからOKと判断する。
